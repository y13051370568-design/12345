# "智算"AI4ML社区平台 - 后端详细设计与实现指南

## 1. 架构总览与技术选型

本系统后端作为整个 AI4ML 平台的核心调度与业务处理中枢，负责承接前端请求、调度多智能体协作、管理云端资源与数据存储。

* **开发语言与框架** ：Python 。使用 FastAPI ，便于快速构建 RESTful API 并处理异步任务。
* **数据库** ：MySQL 。用于持久化用户、任务、社区资源和系统日志等关系型数据。
* **运行环境** ：部署于华为云 Linux 服务器 。

---

## 2. 核心模块与 API 路由设计

基于系统的三大核心角色与四大功能模块，后端 API 应按领域驱动（DDD）或模块化进行拆分，确保边界清晰 。

### 2.1 智能建模任务中心模块 (`/api/tasks`)

面向零基础用户，提供端到端的核心驱动。长时任务需采用异步执行机制，避免阻塞前端 。

* `POST /api/tasks/parse`：自然语言意图解析。接收用户文本，基于规则与关键词识别任务类型（分类/回归）、目标列等 。 **性能约束** ：响应时间需 **≤** 30秒 。
* `POST /api/tasks/upload`：数据集上传。
  * **校验逻辑** ：仅支持 CSV 格式 ；文件大小 **≤** 100MB ；行数 **≤** 10万行 。
* `POST /api/tasks/execute`：触发 Agent 流水线执行建模任务。
* `GET /api/tasks/{task_id}/status`：轮询或 WebSocket 接口，用于前端可视化展示智能体执行进度 。
* `GET /api/tasks/{task_id}/demo`：获取模型的一键 Web 部署测试接口 。

### 2.2 开发者协同优化模块 (`/api/developer`)

面向 AI 开发者，提供 Human-in-the-loop 的干预能力 。

* `PUT /api/developer/tasks/{task_id}/intervene`：提交修改后的智能体决策（如特征工程策略、缺失值处理方式等），并触发后续流水线重跑 。
* `PUT /api/developer/tasks/{task_id}/code`：提交并持久化用户手动修改的 Python 源码，需包含基本语法校验 。
* `GET /api/developer/tasks/{task_id}/download`：打包并导出 Operation Agent 生成的或经过修改的完整 Python 源码及配置文件 。

### 2.3 社区资源与共享复用模块 (`/api/community`)

管理社区生态的流转，所有公开资源均需绑定审核状态 。

* `GET /api/community/{resource_type}`：分页获取已通过审核的公开数据集、模型或工作流 。支持热度、分类、时间筛选。
* `POST /api/community/workflows/{workflow_id}/fork`：Fork 操作。在数据库中深拷贝对应的工作流配置及代码，绑定至当前用户的工作空间 。
* `POST /api/community/{resource_type}/{id}/comments`：提交针对社区资源的评分与评论 。

### 2.4 平台运营管理模块 (`/api/admin`)

需实施严格的 RBAC（基于角色的访问控制），仅限管理员调用 。

* `PUT /api/admin/users/{user_id}/quota`：API Token 额度与预警阈值配置 。
* `PUT /api/admin/audit/{resource_type}/{id}`：处理待审核资源，执行通过、驳回、打标签及设为推荐操作 。资源一旦被执行"下架"操作，立即对外不可见 。
* `GET /api/admin/monitor`：获取资源消耗监控数据，汇总 CPU、内存使用率、并发任务数及 API 调用频率等指标 。

## 3. 数据库物理模型与业务映射

数据库 `ai4ml_community` 的 7 张核心表精确对应了平台的三大业务闭环：用户与资源控制、核心建模状态机、社区分享与复用。

### 3.1 权限与配额底座

* **`sys_user` (系统用户表)**
  * **角色控制** ：通过 `role` 字段严格区分 `ZERO_BASIS` (零基础用户)、`DEVELOPER` (AI开发者) 和 `ADMIN` (管理员)。
  * **资源管控** ：依靠 `api_token_limit` 和 `api_token_used` 实现大模型 API 调用的硬性计费与限流，满足"API额度管理 (ADMIN-002)"需求。

### 3.2 建模流水线与人机协同 (核心引擎)

* **`ml_task` (智能建模任务表)**
  * **状态机运转** ：这是整个后端的枢纽。通过 `status` (CREATED, RUNNING, WAITING_HUMAN, READY_TO_RESUME 等) 和 `current_node` (当前执行节点) 精确控制 Agent 流水线的启停。
  * **上下文文管理** ：`state_json` 存储 Agent 流转的完整上下文，`pending_review_json` 存储需要开发者介入确认的数据，彻底解耦了前端和 Agent 引擎。
* **`ml_task_review` (人机协同审核记录表)**
  * **协同记录** ：支撑"人机协同干预 (DEV-001)"。记录开发者在 `parse_review`、`train_review` 等阶段的干预动作（`action`）和修改的参数（`patch_json`），并利用 `before_version` / `after_version` 实现版本控制与防冲突。

### 3.3 社区生态与资产沉淀

* **资产三剑客** ：`ml_dataset` (数据集)、`ml_model` (模型成果)、`ml_workflow` (开发者工作流)。
* 这三张表共同具备 `is_public` 和 `audit_status` (PENDING, APPROVED, REJECTED) 字段，完美支撑了"资源审核与上下架"的管理员需求。
* **`ml_workflow` (工作流表)**
  * **代码与复用** ：`code_content` 存储持久化后的 Python 源码。`fork_from_id` 记录了开源社区的知识传承路径，支撑"Fork复用 (DEV-004)"。
* **`community_comment` (社区评论表)**
  * 通过 `resource_type` (DATASET, MODEL, WORKFLOW) 实现多态关联，统一管理平台所有的评分与反馈互动。

## 4. 关键非功能性约束落地方案

### 4.1 并发与资源控制 (核心难点)

* **任务队列**：为满足"同一时刻最多 5 个任务并行执行"的硬性约束，后端必须引入任务队列（如 Celery 配合 Redis，或 Python 原生的 `asyncio.Queue` / 线程池）。所有新建任务进入排队，工作进程池（Worker Pool）容量限制为 5。
* **并发访问**：支持 50 个用户同时在线，需要确保后端的 Web 框架（如 uvicorn/gunicorn）配置合理的 worker 数量和连接池大小。

### 4.2 安全与数据保护

* **隔离策略**：普通用户仅可访问和管理自己工作空间内的任务、数据和记录。在所有业务层的查询和修改接口中，必须校验 `user_id`。
* **凭证防泄露**：外部大模型 API Key、数据库密码、云服务器密钥必须通过环境变量或独立的配置文件（如 `.env`）挂载，绝对禁止硬编码在代码仓库中。

### 4.3 可观测性与异常处理

* **分级错误提示**：当底层发生异常（如 解析 CSV 失败，或 训练发散），后端不应直接将异常堆栈抛给前端，而是需要进行捕获（Try-Catch），并转化为面向业务用户的可理解提示（如"文件格式错误"或"特征列包含无法处理的空值"）。
* **日志持久化**：长时任务执行时，需记录每个阶段的时间戳和资源消耗。即便单个节点失败，也要确保整条任务链路的状态可查。
